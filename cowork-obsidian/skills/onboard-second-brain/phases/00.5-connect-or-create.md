# Phase 0.5 — Connect Existing OR Create New

**Goal:** Branch the install based on whether the user already has an Obsidian vault. Auto-detect existing vaults from Obsidian's own config. If yes — connect to one (without reorganizing it). If no — fall through to Phase 1 to scaffold a fresh vault.

**Time:** ~5–10 minutes (much shorter than the create-new path).

---

## Script — what to say (in voice)

> *"Phase 0.5 is the fork in the road.*
>
> *Two paths:*
>
> *(A) You already have an Obsidian vault — somewhere on this machine, that you opened in Obsidian at least once. I'll auto-detect it from Obsidian's own config and offer to connect. We do **not** reorganize your vault; we don't impose any folder structure on you.*
>
> *(B) You don't have a vault yet, or you want a fresh dedicated one for the second-brain workflow. We scaffold from scratch in Phases 1–3.*
>
> *Don't worry about picking yet. I'll auto-detect first, then ask."*

---

## Step 1 — Detect OS

```
mac     → obsidian_config_path = ~/Library/Application Support/obsidian/obsidian.json
windows → obsidian_config_path = %APPDATA%\obsidian\obsidian.json   (= C:\Users\<u>\AppData\Roaming\obsidian\obsidian.json)
linux   → obsidian_config_path = ~/.config/obsidian/obsidian.json
```

Capture the OS in state file. We'll reuse it in Phase 3 (if we end up scaffolding) and Phase 7 (if user opts into Desktop MCP).

---

## Step 2 — Try auto-detection

Read `obsidian_config_path`.

```
If file missing                  → step 4 (manual fallback — Obsidian probably never installed)
If file exists but JSON-parse fails → step 4 (manual fallback)
If file exists, parse OK        → step 3
```

The file structure (verified against real Obsidian installs as of 2026-05-09):

```json
{
  "vaults": {
    "<random-id-1>": { "path": "<absolute path>", "ts": <unix-ms>, "open": true },
    "<random-id-2>": { "path": "<absolute path>", "ts": <unix-ms> }
  }
}
```

For each vault entry:
- `path` is the absolute folder path (Windows uses forward or back slashes — both are valid in the JSON)
- `ts` is the last-opened timestamp
- `open` is `true` for the currently-open vault, omitted otherwise

---

## Step 3 — Present detected vaults

For every entry parsed in step 2:
- Compute display name = `basename(path)`
- Verify the path actually exists on disk (drop stale entries with a one-line note: *"Note: vault `<name>` listed but folder missing on disk — skipping."*)
- Mark the entry with `"open": true` as **(currently open)**

Show the user a numbered list. Example:

```
I found these Obsidian vaults on your machine:

  1. Hermes Second Brain      — D:\dev\Hermes Second Brain\Hermes Second Brain
  2. Paperclip               — D:\dev\Paperclip - Second Brain\Paperclip
  3. Claude Code - Second Brain  (currently open)  — D:\dev\Claude Code - Second Brain
  4. Open Claw - Second Brain — D:\dev\Open Claw - Second Brain\Open Claw
  5. Obsidian Second Brain   — D:\dev\Obsidian Second Brain\Obsidian Second Brain

Pick one (1–5), or:
  z) None of these — let me paste a path manually
  n) Skip — let's create a brand-new vault from scratch (Phase 1 onward)
```

If the user picks 1–N → step 5 (register the vault).
If the user picks `z` → step 4 (manual paste).
If the user picks `n` → tell the user *"Got it — moving to Phase 1 to plan a fresh vault."* Set `vault_mode_default: scaffold` in state and advance.

---

## Step 4 — Manual fallback

Use this when auto-detect fails OR the user picks `z`.

> *"I couldn't auto-detect (or you chose manual). Three options:*
>
> *(a) Open Obsidian first, create or open a vault, then run `/onboard-second-brain` again — I'll pick it up from `obsidian.json`.*
> *(b) Paste the absolute path to your existing vault folder right now (e.g., `/Users/sarah/MyVault/` or `D:\Knowledge\PARA\`).*
> *(c) Skip and let's create a new vault from scratch (Phase 1 onward)."*

If option (b): user pastes path. Validate:
- Path exists on disk?
- Path contains an `.obsidian/` folder? (proves it's a real Obsidian vault, not just any folder)

If both checks pass → step 5. If checks fail, explain which one and ask again.

If option (a) or (c) → set state appropriately and advance to next phase.

---

## Step 5 — Register the existing vault

Add the picked vault to `projects/second-brain/vaults.md` with `mode: external`.

```markdown
### <vault-name-from-basename>
- name: <slug-of-basename>          # lowercase, hyphens
- path: <absolute path with forward slashes>
- mode: external
- purpose: <ask user — one short sentence; optional, can be filled in later>
- life_areas: []                    # empty for external — we don't impose structure
```

Show the diff to the user before saving. Ask permission. Save.

Set in state file: `vault_mode_default: external`.

---

## Step 6 — Offer the overlay option

> *"You're connected in `external` mode — I won't touch your vault structure. I'll read everything (except `.obsidian/` and `about-me/`) and only write to paths you explicitly name in commands.*
>
> *Optional: do you want me to add a `raw/wiki/output` overlay on **one sub-folder** of this vault? This gives you the second-brain workflow on that one slice without affecting the rest of your vault.*
>
> *For example, if your vault is `MyVault/` and you have a folder `MyVault/coaching/`, I can add `MyVault/coaching/raw/`, `MyVault/coaching/wiki/`, `MyVault/coaching/output/` — and the `/second-brain build coaching` prompt would work on just that area.*
>
> *Yes / no?"*

If yes:
1. Ask which sub-folder. Confirm it exists.
2. Validate it's NOT the vault root (overlay must be a sub-folder).
3. Create the three folders: `<sub-folder>/raw/`, `<sub-folder>/wiki/`, `<sub-folder>/output/`. Ask permission before each batch.
4. Update the vault entry in `vaults.md`:
   ```markdown
   - mode: overlay
   - overlay_root: <absolute path to chosen sub-folder>
   - life_areas: [<basename-of-sub-folder>]
   ```
5. Set state: `vault_mode_default: overlay`.

If no: leave the vault as `mode: external`. Done.

---

## Step 7 — Skip Phases 1–3

The user is connected to an existing vault, so the vault-planning, Obsidian-install, and scaffold phases don't apply. Update state:

- `current_phase: 4` (jump straight to wire-Cowork-to-vault)
- `Phase 0.5 (connect existing): completed at <ISO timestamp>`
- `Phase 1 (plan vaults): skipped — existing vault used`
- `Phase 2 (install obsidian): skipped — vault already exists, app already used`
- `Phase 3 (scaffold first vault): skipped — connecting to existing` (or, if overlay was added, `Phase 3 (scaffold first vault): partial — overlay folders created at <path>`)

Append to `projects/second-brain/memory.md`:

```markdown
## <ISO timestamp> — Phase 0.5 complete: connected to existing vault
- Vault: <name> at <path>
- Mode: <external | overlay>
- Overlay root (if overlay): <path>
- Skipped: Phases 1–3 (existing vault, no scaffold needed)
```

---

## If user chose "create new" (path B)

Just advance to Phase 1 with `vault_mode_default: scaffold`. The existing Phases 1 → 2 → 3 → 4 → 5 flow handles everything.

---

## Verification before advancing

- `vaults.md` has at least one vault entry with a valid `mode` field.
- For `external`: `path` exists on disk and contains `.obsidian/` folder.
- For `overlay`: the overlay sub-folder exists and now has `raw/`, `wiki/`, `output/` subfolders.
- For `scaffold` (path B users): nothing yet — Phase 3 will populate.
- State file shows `vault_mode_default` set.

---

## What this phase never does

- Reorganize, move, or rename anything inside an existing vault (mode = external is **read-everything, write-nothing-without-explicit-naming**).
- Force the raw/wiki/output overlay on the vault root (overlay must be a sub-folder).
- Touch the `.obsidian/` folder (Obsidian's own settings — off-limits).
- Touch any `about-me/` folder that already exists in the vault (those belong to the user's existing setup).
- Auto-detect on a machine where Obsidian was never opened (returns 0 vaults — fall through to manual or scaffold).

---

## Mac note: first-access permission prompt

The first time Cowork tries to read a vault outside the user's home Documents folder on macOS, the OS shows a "Allow Cowork to access this folder?" prompt. Pre-warn the user:

> *"Heads up: macOS may ask you to allow Cowork to access this folder. Click Allow — that's the OS confirming you're letting me read it."*

No equivalent prompt on Windows or Linux for user-owned folders.
