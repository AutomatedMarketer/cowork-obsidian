## 2 🧠 THE BIG IDEA (READ THIS FIRST — IT'S SHORT)

There are two completely different things that "sync" can mean. They're easy to confuse and one of them is dangerous:

1. **Sync the vault folder bytes via cloud-storage tools (Dropbox / iCloud / OneDrive / Google Drive).** ❌ Don't. Cloud-storage services auto-sync the folder bytes underneath the running Obsidian and Cowork apps. They cause duplicate files (`Note.md` and `Note 1.md`), corrupted writes (partial syncs while a write is in flight), and phantom disappearing content. We forbid this in Lesson 1, and Lesson 3 doesn't relax that rule.

2. **Sync at a different layer, designed for live-edited files.** ✅ Three approved options:
   - **Obsidian Sync** is application-layer — Obsidian itself manages encryption + transfer through Obsidian's own SaaS service.
   - **Syncthing** is peer-to-peer — direct device-to-device sync with conflict awareness designed for live-edited files. No cloud server involved.
   - **Git/GitHub** is version control — every save is an explicit commit, full history, and you push when you're ready.

Pick one. They're all good. The differences are about *how you want to pay for it, where you want your data to live, and whether you also need mobile.*

The **Optional Challenge** at the end of this lesson is a separate concept: connecting Claude **Desktop** (the chat app, NOT Cowork) to the same vault via an MCP server. That's about getting *two Claude products* to see the same vault on one machine, not about syncing one vault across multiple machines.

## 4 🤖 OPTION A — OBSIDIAN SYNC (PAID, EASIEST)

> *Recommended cohort default.*

### What it is

Obsidian Sync is Obsidian's official paid SaaS service. End-to-end encrypted (AES-256), Cure53 audited, mobile included.

**Pricing (as of 2026-05-09):**

| Tier | Price | What you get |
|---|---|---|
| Sync Standard | $4/mo annual or $5/mo monthly | 1 vault, 1 GB storage, 5 MB file cap, 1-month version history |
| Sync Plus | $8/mo annual or $10/mo monthly | 10 vaults, 10 GB → upgradable to 100 GB, 200 MB file cap, 12-month history |

Both tiers include end-to-end encryption, offline access, file recovery, selective sync (skip large media if you want), and one subscription covers iOS + iPadOS + Android.

### Two rules you MUST understand before you start

1. **The first device you enable Sync on becomes the source of truth** for the remote copy. We do this on your **main / most-used machine first**. If you set it up on a laptop and then add your desktop, the desktop will sync DOWN the laptop's vault state — which is probably not what you want. Pick the machine where the vault is most complete now, and use it first.

2. **The end-to-end encryption password is unrecoverable.** If you lose it, you lose access to your synced vault on every new device — by design. Cure53 audited; AES-256. We'll set the password and store it in your password manager (1Password, Bitwarden, etc.) **immediately**, before going further.

### Setup (Mac + Windows — identical steps)

The steps are identical on both OSes. Cowork will walk you through them in Phase 6, but here's the manual sequence so you know what to expect:

1. Open Obsidian on your **main / most-used** machine (do NOT do this on a secondary machine first — see Rule 1 above).
2. **Settings → Core plugins → Obsidian Sync** → toggle on.
3. **Settings → Obsidian Sync** → **Sign in** to your Obsidian account. (Free Obsidian account required first; sign up at https://obsidian.md/account if you don't have one.)
4. **Choose or create a Sync subscription** — Standard or Plus.
5. **Set the End-to-End Encryption password.** Pause here. Do this:
   - Open your password manager
   - Generate or pick a strong password
   - Save it under entry name *"Obsidian Sync E2EE — `<vault name>`"*
   - Note in the entry: *"Unrecoverable. Used to encrypt vault contents before they leave my device."*
   - **Confirm saved before pasting into Obsidian.** This is the one place you cannot afford a typo.
6. **Choose the vault to sync** — the vault you set up in Lesson 1.
7. Wait for the initial upload (1 GB Standard cap, 5 MB per file).

### On the second device

1. Install Obsidian (Lesson 1 covers this if needed).
2. Open Obsidian → Settings → Core plugins → enable **Obsidian Sync** → Settings → Obsidian Sync → **Sign in** (same account).
3. **Choose "Receive a vault"** (NOT "Send" — sending would overwrite the remote with this device's empty vault).
4. Pick the same vault name → Obsidian downloads it locally.
5. The E2EE password decrypts on this device — paste from the password manager.
6. Done. Edits on either machine sync to the other within seconds when both are online.

```

After Step 7 succeeds and the initial upload finishes, append a one-line entry to

(Standard or Plus), date.

Don't tell me to add a second device today — we'll do that separately when I'm at

```

## 5 🤖 OPTION B — SYNCTHING (FREE, PEER-TO-PEER)

### What it is

Syncthing is a free, open-source, peer-to-peer file sync tool. No cloud, no monthly fee, files never enter someone else's servers. The trade-off: per-device pairing setup (~5 min per device) and mobile support is harder.

**Best fit if:** you have 2–3 desktops/laptops, you don't need mobile, you're comfortable poking at a web UI for setup, and you want everything to stay on your machines.

### Mac install

```
brew install syncthing
brew services start syncthing
```

Then open `http://localhost:8384` in a browser.

### Windows install

1. Download from https://syncthing.net/downloads/
2. Run the installer. Pick the **service** install option for auto-start.
3. Open `http://localhost:8384` in a browser.

### Pairing devices

On both devices:

1. Note the **Device ID** (in Syncthing's web UI: Actions → Show ID).
2. On device 1: Add Remote Device → paste device 2's ID → name it.
3. On device 2: Accept the incoming pairing request.

### Add the vault folder

On device 1:

1. Add Folder → point at the vault path → set Folder ID (anything memorable).
2. Share with the device you just paired.

On device 2:

1. Accept the incoming folder share.
2. Set the local path where the vault should live on this machine.
3. Sync starts immediately.

### Conflict resolution

Syncthing handles conflicts at the file level — typically renames one copy with a `.sync-conflict-...` suffix. Less elegant than Obsidian Sync, but rare in practice if devices are connected often.

### The paste-ready prompt

```
I'm on Lesson 3 of Project 04 — Second Brain. I picked Syncthing as my sync method.

to do both devices in one session.

After the first device is configured, append a one-line entry to
projects/second-brain/memory.md noting: Syncthing enabled, vault path, primary
```

## 6 🤖 OPTION C — GIT / GITHUB VIA OBSIDIAN GIT PLUGIN (FREE, WITH HISTORY)

### What it is

Auto-commits your vault to a private GitHub repo. Free, full version history, every save is a commit you can roll back from. Works on every desktop.

**Best fit if:** you already use GitHub for other things, want version control as a side benefit, and don't mind a one-time CLI step.

**Mobile caveat:** the Obsidian Git plugin uses isomorphic-git (JavaScript) on mobile, has memory constraints, and no SSH support. Don't rely on it as your primary mobile path.

### Step 6.1 — Create the private GitHub repo

```
gh repo create <username>/my-vault --private --description "My Obsidian second brain — synced via Obsidian Git"
```

Replace `<username>` with your GitHub handle. Note the resulting URL.

### Step 6.2 — Install the Obsidian Git plugin

In Obsidian:

1. Settings → Community plugins → Browse.
2. Search for **"Obsidian Git"** (author: denolehov).
3. Install + enable.

### Step 6.3 — Configure the plugin

Settings → Obsidian Git:

- **Vault backup interval (minutes):** 10 (or whatever cadence you prefer)
- **Auto pull interval (minutes):** 10
- **Pull updates on startup:** on
- **Commit message:** `vault auto-commit {{date}}` (or leave default)
- **Disable push:** off
- **Disable pull:** off

### Step 6.4 — Initialize the local git repo

In a terminal at the vault path:

```bash
cd <vault-path>
git init
git remote add origin git@github.com:<username>/my-vault.git
git add .
git commit -m "initial vault"
git push -u origin main
```

(If using HTTPS instead of SSH, the URL is `https://github.com/<username>/my-vault.git`.)

### Step 6.5 — Test the loop

1. Edit a note in Obsidian.
2. Wait for the auto-commit interval (or trigger manually: Command palette → "Obsidian Git: Commit all changes").
3. Check `https://github.com/<username>/my-vault/commits/main` — the commit should appear.

### Step 6.6 — Set up the second machine

```bash
gh repo clone <username>/my-vault <vault-path-on-this-machine>
```

Then in Obsidian on this machine: open the cloned folder as a vault. Install + enable Obsidian Git. Same plugin config as device 1.

### The paste-ready prompt

```

5. Test by editing a note, waiting for an auto-commit, and confirming the commit

After Step 5 succeeds, append a one-line entry to projects/second-brain/memory.md
noting: Git/GitHub sync enabled, repo URL, auto-commit interval, date.

other machine.
```

## 8 💪 OPTIONAL CHALLENGE — CLAUDE DESKTOP MCP

> **This is genuinely optional.** Skip if you only use Cowork. Most cohort members can finish Lesson 3 here and never come back.

### Why this exists (and the trap to avoid)

Claude Cowork already reads your vault directly via filesystem — that's how Lesson 1 Phase 4 wired things up. Cowork doesn't need MCP.

But **Claude Desktop** (the standalone chat app you launch from your Dock or Start menu) is a different product. It has *no native filesystem access by default.* If you want Claude Desktop to also read your vault for chat queries, you need to install an MCP server.

**The trap that wastes everyone's afternoon:** if you Google how to install an MCP for Claude Desktop on Mac, most guides will tell you to edit one of these files:

| File | Result |
|---|---|
| `~/.claude/settings.json` | ❌ DOES NOT WORK for Claude Desktop. That file is for Claude Code (Cowork) only. |
| `~/.claude/mcp.json` | ❌ DOES NOT WORK either. Claude Desktop tries an OAuth flow on this and fails for non-OAuth servers. |
| `enabledMcpjsonServers` in some settings | ❌ Also doesn't work. |

The **only** file Claude Desktop reads for MCP server config is:

- **Mac:** `~/Library/Application Support/Claude/claude_desktop_config.json`
- **Windows:** `%APPDATA%\Claude\claude_desktop_config.json` (= `C:\Users\<you>\AppData\Roaming\Claude\claude_desktop_config.json`)

Anything else is a dead end. Cohort note from the GoHighLevel MCP setup confirmed this exact same trap — it cost real hours before we figured it out. Don't repeat that.

### Step 8.1 — Check Node.js is installed

The MCP server we'll use ([`iansinnott/obsidian-claude-code-mcp`](https://github.com/iansinnott/obsidian-claude-code-mcp)) uses WebSocket auto-discovery and requires Node.js (npx).

```
node --version
```

Expected: `v18.x` or later. If missing or older:

- **Mac:** `brew install node` (if Homebrew is installed) OR download from https://nodejs.org
- **Windows:** Download from https://nodejs.org and run the installer

### Step 8.2 — Install the Obsidian-side plugin

Open Obsidian. **Settings → Community plugins → Browse → search for "Claude Code MCP"** (author: iansinnott) → install + enable.

This plugin opens a WebSocket on port 22360 (avoids common dev-server conflicts). It exposes the vault to MCP clients.

### Step 8.3 — Quit Claude Desktop FULLY

- **Mac:** Cmd+Q on the Claude Desktop window (or right-click the menu bar icon → Quit). Don't just close the window — fully quit.
- **Windows:** Right-click the system tray icon → Quit. Confirm via Task Manager that no Claude Desktop process remains.

If you skip this step, the next step's edits won't take effect.

### Step 8.4 — Edit `claude_desktop_config.json`

Mac path: `~/Library/Application Support/Claude/claude_desktop_config.json`
Windows path: `%APPDATA%\Claude\claude_desktop_config.json`

If the file doesn't exist, create it. If it exists, merge the `obsidian` entry into the existing `mcpServers` object:

```json
{
  "mcpServers": {
    "obsidian": {
      "command": "npx",
      "args": ["-y", "@iansinnott/obsidian-claude-code-mcp-bridge"]
    }
  }
}
```

> **Note for users with existing MCP servers (e.g., GoHighLevel):** if `mcpServers` already has entries, just add the `obsidian` key as a sibling. Don't replace the whole object.

Save the file.

### Step 8.5 — Open Claude Desktop and verify

1. Open Claude Desktop fresh (it must have been fully quit in Step 8.3).
2. In Claude Desktop: click your name (top right) → **Customize** → **Connectors**.
3. Scroll to the **Desktop** section.
4. You should see **obsidian** listed and connected ✅.

### Step 8.6 — Test in Claude Desktop

Ask Claude Desktop something that requires reading your vault:

> *"Read my notes about [topic from one of your wiki pages]. Quote the wiki page directly."*

Claude Desktop should respond with content from the wiki page, citing the file path.

### The paste-ready prompt

> **Important:** This optional challenge is half manual (editing Claude Desktop's config file is something Cowork can guide but the user has to do themselves outside Cowork). Cowork CAN write to the config file if you point it there with proper permissions, but the safer path is to do it manually with Cowork narrating.

```
I'm doing the optional Lesson 3 challenge — wiring Claude Desktop to my vault via
MCP. Walk me through Sections 8.1 through 8.6 of the Lesson 3 SOP, pausing for my

Specifically:

1. Verify Node.js is installed (node --version).
2. Tell me to install the Obsidian Community plugin "Claude Code MCP" by
   iansinnott, and pause until I confirm.
3. Tell me to FULLY QUIT Claude Desktop (Cmd+Q on Mac, right-click tray icon on
   Windows) and pause until I confirm.
4. Show me the exact JSON to paste into ~/Library/Application Support/Claude/
   claude_desktop_config.json (Mac) or %APPDATA%\Claude\claude_desktop_config.json

   replacing them. Pause until I confirm I've saved the file.
5. Tell me to reopen Claude Desktop and check Connectors.
```

## 11 🛡 SAFETY NOTE

Three rules. Each one prevents real data loss or wasted hours.

1. **Cloud-storage folder sync is still forbidden.** Lesson 1 forbade it; Lesson 3 doesn't relax that rule. The three sync options in this lesson are application-layer (Obsidian Sync), peer-to-peer (Syncthing), or version-control (Git) — *none of them auto-sync the folder bytes underneath the running app.* Don't move your vault into iCloud / OneDrive / Dropbox / Google Drive thinking it'll be "easier sync." It corrupts.

2. **First-device-wins for Obsidian Sync.** Set it up on your main machine first. The vault state on that machine becomes the remote source of truth. If you set it up on a less-complete device first, you'll sync that less-complete state down to your main machine and lose work.

3. **The Obsidian Sync E2EE password is unrecoverable.** If you lose it, your vault on every new device is gone. Save it in a password manager AT THE MOMENT you create it, not "later." Same for any critical secrets in the Git/GitHub option (SSH keys, etc.).

## 12 🔍 UNDER THE HOOD — FOR THE CURIOUS (TOTALLY OPTIONAL)

Skip this unless you want to know why three options.

### Why these three and not others

The three approved options cover three different "I want sync because..." reasons:

- *"I want it everywhere, including mobile, with zero friction"* → Obsidian Sync (the official, paid, easy path).
- *"I want it free, on my devices only, and not on someone else's servers"* → Syncthing (peer-to-peer, you control the wire).
- *"I want it free, with full history, and I already use GitHub"* → Git/GitHub (version control as sync).

There's no fourth option that covers a meaningfully different reason. Anything else is a variation of these three. (E.g., Obsidian's competing services like Logseq Sync are the same shape as Obsidian Sync. Self-hosted CouchDB is a Syncthing variant. Other DVCS options are git variants.)

### Why we don't recommend "just zip the vault to OneDrive"

Cloud-storage services (OneDrive, iCloud, Dropbox, Google Drive) auto-sync the folder bytes — they treat your vault as if it were a Word doc that's only being edited by one person. When Obsidian or Cowork is in the middle of writing a markdown file (which can take milliseconds for big notes with images), the cloud-storage sync grabs a half-written byte stream and uploads it. The other device pulls that half-written stream. Result: a corrupt note. Or worse — a cloud-storage conflict copy where neither version is what you want.

The three approved options are aware of file write atomicity. Obsidian Sync is built for live-edited notes. Syncthing has a versioning + conflict-detection system designed for exactly this case. Git literally creates an explicit snapshot at commit-time so there's no "half-saved file in the wire" condition.

### Why Phase 7 (Desktop MCP) is separate from sync

Sync = "same vault, multiple computers."
Desktop MCP = "same vault, multiple Claude products on one computer."

These are orthogonal. You might want both, or just one, or neither. Cohort path is Cowork-only — most students will never need Phase 7. Power users who chat in Claude Desktop and code in Cowork get both products talking to the same vault — that's the value.

### What MCP actually is (1 paragraph)

MCP (Model Context Protocol) is Anthropic's standard for letting AI clients (like Claude Desktop) talk to external tools and data sources. An MCP server is a small program that exposes a specific capability — read a folder, query a database, post to Slack — through a standardized protocol. Claude Desktop has no built-in filesystem access; you grant it filesystem access by running an MCP server that does. The Cowork product (Claude Code) has filesystem access built in, so it doesn't need MCP for files. Same engine, different product surface.
