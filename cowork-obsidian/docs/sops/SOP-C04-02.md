## 1 🎯 WHAT THIS LESSON GIVES YOU

By the end of this lesson, one of your life areas (or one slice of your existing vault, in `external`/`overlay` mode) has a real wiki — multiple short pages, each on a single topic, cross-linked, built by Cowork using the `/second-brain` skill.

You'll know **the four operations** of `/second-brain`:

- `/second-brain build [area]` — first-pass wiki construction (run once per area).
- `/second-brain update [area]` — fold new raw notes into the existing wiki (run weekly).
- `/second-brain health-check [area]` — flag contradictions, gaps, stale pages (run monthly).
- `/second-brain fold-back [area]` — absorb finished outputs back into the wiki (run quarterly).

**The big shift in v0.3.0 vs paste-the-prompt versions:** you don't paste prompt text anywhere. The plugin already installed the `/second-brain` skill; you just invoke it. The skill knows the prompts, the rules, the modes, the safety checks. You type a slash command and the skill does the work.

---

## 2 🧠 THE BIG IDEA (READ THIS FIRST — IT'S SHORT)

Every operation on your wiki goes through `/second-brain`. You don't touch wiki pages by hand. You don't write prompts. You write into `raw/` freely, then you run a slash command, and the skill does the rest.

**Three rules built into the skill that prevent 90% of confusion:**

1. **Build is destructive. Update is safe.** `build` overwrites existing wiki pages. `update` preserves them and adds. **Run `build` once per area — the first time. Every refresh after that is `update`.** The skill warns you if you try to run `build` on an area that already has a wiki.
2. **The skill never writes to `raw/`.** Raw is your space. It reads raw, it doesn't edit raw.
3. **The skill is mode-aware.** In `scaffold` mode it operates silently inside `<area>/{raw,wiki,output}/`. In `overlay` mode it's scoped to your overlay sub-folder. In `external` mode it asks you per-write where to put things — your existing vault structure stays your call.

## 3 🤖 DO THE THING — THE EASY WAY (RECOMMENDED)

### Step 1 — Open Cowork at your vault root (or workspace)

If you haven't already, point Cowork at the folder containing `projects/second-brain/vaults.md`. Usually that's your Cowork workspace from Project 01, or the vault root itself if you went standalone.

(Or just run `/open-vault` from any Cowork session — it loads vault context for you.)

### Step 2 — Run `build` for the first time

Type one of these into Cowork:

```
/second-brain build
```

Or with an explicit area (recommended):

```
/second-brain build coaching
```

(Replace `coaching` with whatever life area you seeded raw notes in, from Lesson 1. If you skip the area name, the skill picks the area with the most raw material and asks you to confirm.)

The skill will:

1. Load vault config (read `vaults.md`, identify mode).
2. Verify the target — for `scaffold` it picks the life area; for `overlay` it uses the overlay sub-folder; for `external` it asks you which folder to read.
3. Read every file in the target's `raw/` folder.
4. Read your `about-me/` files for tone and context.
5. Synthesize topics — one wiki page per topic, cross-linked via `[[wikilinks]]`.
6. Show you the drafts before saving (in `external` mode, asks per-page).
7. Save approved pages to the target's `wiki/` folder.
8. Append a log entry to `projects/second-brain/memory.md`.

**That's the whole `build` operation.** No prompt to copy. The skill has the prompt baked in.

If anything looks off in the drafts, tell the skill: *"page 3 lost the cross-link to page 5, please fix"* — and it will. You're conversing with the skill, not babysitting a paste operation.

### Step 3 — Add a new raw note (small one)

In Obsidian (or any text editor), drop a new note into the same area's `raw/` folder. Name it with today's date prefix:

```
<area>/raw/2026-05-09-quick-observation.md
```

Two or three sentences is enough. Something you noticed today that belongs in this area.

### Step 4 — Run `update`

```
/second-brain update coaching
```

The skill:

1. Reads the existing wiki for this area to understand what's already covered.
2. Reads only raw notes newer than the last update timestamp.
3. For each new note:
   - If it extends an existing wiki page → revise that page (preserve unchanged sections).
   - If it's a new topic → create a new wiki page, cross-link from related existing pages.
   - If it contradicts an existing page → adds a "⚠ Contradiction noted" section to the affected page; surfaces it for your review (does NOT silently overwrite).
4. Tells you exactly which pages were modified or created.

You'll usually see *"updated 1 page, created 0 new pages"* on a small change like the one in Step 3. That's correct — it shouldn't churn pages that didn't need to change.

### Step 5 — Run `health-check`

```
/second-brain health-check coaching
```

The skill reads the entire wiki for the area and reports:

- **Contradictions** — places where two wiki pages disagree.
- **Missing topics** — topics referenced in raw but never developed into a wiki page.
- **Stale pages** — wiki pages with no recent raw activity (e.g., last raw note on this topic is 6+ months old).
- **Orphans** — wiki pages with no incoming or outgoing wikilinks.
- **Output → wiki gaps** — outputs in `output/` that contain ideas not yet folded back into the wiki.

The report saves to `<wiki>/_health-check-<YYYY-MM-DD>.md` and also prints inline so you can skim it.

You read the report. You pick what to act on. The skill never silently "fixes" things — it surfaces, you decide.

### Step 6 — Skip `fold-back` for now

`fold-back` is the fourth operation — it absorbs finished outputs from `output/` back into the wiki. You don't have any outputs yet (those come from real Cowork sessions over time), so we cover this one in Lesson 4 when you're setting up the quarterly cadence.

## 4 ✅ MAKE SURE IT WORKED (3 MINUTES)

Three checks.

### Check 1 — The wiki folder has multiple pages, not one mega-page

In Obsidian, navigate to the area's `wiki/` folder. You should see 3–15 separate `.md` files, each on a distinct topic. Titles visible. Each file short (a page or two max).

If you got one giant file instead → tell the skill:

> *"`/second-brain build` produced one mega-file instead of separate pages. Re-run with explicit instruction to create one page per topic. Delete the mega-file before re-running."*

### Check 2 — Cross-links exist and work

Open any wiki page in Obsidian. Look for `[[Another Page Name]]` links. Click one — it should jump to the referenced page.

If there are no cross-links → tell the skill:

> *"My wiki pages don't have cross-links between them. Re-run `/second-brain update` and add cross-links between any pages that relate to each other's topics."*

### Check 3 — Health-check report exists

Find `<wiki>/_health-check-<today>.md` in your file manager or Obsidian. It should have a list of contradictions, missing topics, pages needing updates, and pages suggested to add.

If it's missing → re-run `/second-brain health-check` and confirm the file landed.

## 7 ⚠ IF SOMETHING DOESN'T WORK

| WHAT YOU SAW | WHAT TO SAY TO COWORK |
|---|---|
| The wiki folder is empty after `build` | *"`/second-brain build` ran but the wiki folder is empty. Tell me where you actually wrote the files and walk me through retrying with the explicit target path."* |
| One giant `wiki.md` instead of many pages | *"`/second-brain build` produced one mega-file. Delete it, re-run with explicit instruction that I want one page per topic — no consolidation."* |
| `update` rewrote pages I liked | *"`/second-brain update` changed pages I didn't want changed. Restore the prior versions if possible (or tell me what got modified) and tighten the operation so update is more conservative — only touch pages where new raw material justifies a change."* |
| Wiki pages don't sound like me | *"The wiki pages don't sound like me — they sound generic. Verify you read `about-me/About Me.md` and `about-me/Brand Voice.md` first. If you did, tell me what to add to Brand Voice to make the wiki feel more mine."* |
| `build` warned me about overwriting an existing wiki | *"You're right — I shouldn't `build` an area that already has a wiki. Cancel the `build` and run `/second-brain update` instead so I don't lose the existing pages."* |
| The skill won't run because it doesn't see `vaults.md` | *"`/second-brain` says `vaults.md` is missing. Walk me through verifying the file exists at `projects/second-brain/vaults.md` and that Cowork is pointed at the right workspace."* |
| External mode kept asking "which folder?" | *"In external mode the skill asks me which folder every time — that's expected (no auto-creation in external mode). But cache my last answer for this session so I don't have to re-answer mid-lesson."* |
| I want to see the raw prompt the skill uses | *"Show me the exact prompt template `/second-brain build` is running internally so I can understand what's happening under the hood. (Reference appendix in SOP-C04-02 if you have it.)"* |

The golden rule: **describe what you see, ask the skill to fix it.**

## 8 🛡 SAFETY NOTE

Two rules to keep the wiki trustworthy:

1. **Never run `build` on an area that already has a wiki.** Build overwrites. The skill warns you, but you can override the warning by mistake. If you want to refresh an existing wiki, use `update`. If the wiki is broken and you want to rebuild, copy `wiki/` → `wiki-backup-<date>/` first, THEN run `build`.

2. **Back up the vault.** It's a folder of `.md` files. If you set up Git/GitHub sync in Lesson 3, you already have automatic backups (every commit is a snapshot). Otherwise, zip it weekly or commit it to a private Git repo.

## 9 🔍 UNDER THE HOOD — FOR THE CURIOUS (TOTALLY OPTIONAL)

Skip this unless you want to know what's happening behind the slash command.

### What the skill actually does internally

When you run `/second-brain build coaching`, the skill (`cowork-obsidian/skills/second-brain/SKILL.md`) loads. The skill is a markdown file with structured instructions. Cowork reads those instructions and executes them — load `vaults.md`, find the target, read raw, synthesize topics, write wiki pages, log to memory. The "prompt" is baked into the skill file.

This is the difference between v0.1.0 (where you pasted the prompt yourself) and v0.3.0 (where the plugin owns the prompt). Same logic, less friction.

### Why three (well, four) operations and not six

The operations map to four distinct intents:

- **Create** — `build`
- **Refresh** — `update`
- **Audit** — `health-check`
- **Absorb** — `fold-back`

Anything else fits inside one of those. *"Summarize my wiki"* = `health-check` with a summary at the end. *"Restructure my pages"* = `update` with a directive. *"Compress page X"* = an inline edit you ask the skill to do without invoking a top-level operation.

Four operations is the minimum that covers the workflow. More operations = more cognitive overhead = people stop using them.

### Why mode-awareness matters

In `scaffold` mode the paths are predictable: `<area>/raw/` → `<area>/wiki/`. The skill runs silently. In `external` mode the paths are whatever you chose — the skill can't guess, so it asks. Trade-off: friction for safety.

If you find external mode too slow for a frequent target, you can add an `overlay` for that one slice (re-run `/onboard-second-brain` and add an overlay on the sub-folder you care about). Then operations on that sub-folder become silent again.

## 10 📎 APPENDIX — THE RAW PROMPT TEMPLATES (FOR REFERENCE / DEBUGGING ONLY)

You don't paste these. The skill already runs them. They're documented here only so you can:

- Understand exactly what Cowork is doing under the hood.
- Debug if the skill produces unexpected output (you can compare what it ran vs. what it should have).
- Reuse them in a different AI tool (e.g., paste into ChatGPT or another assistant) if you ever want to swap providers.

### Build template

```
Read every file inside of <RAW-INPUT-FOLDER>. Create one wiki page per major topic
you find across the files. Write each wiki page as a clean markdown file with a
title, a two-to-three-sentence summary, key points, and cross-links to related
pages using [[wikilink]] syntax. Save every file into <WIKI-OUTPUT-FOLDER>. Do
not summarize multiple topics into one file — create separate pages per topic.
```

### Update template

```
Read every file in <RAW-INPUT-FOLDER> and check the wiki at <WIKI-OUTPUT-FOLDER>.
Update any wiki pages that have new relevant information from raw, add new wiki
pages for any topics that aren't yet covered, update all cross-links, add keyword
links where relevant. Do NOT delete any existing pages. Only add and improve.
```

### Health-check template

```
Read all the wikis in <WIKI-OUTPUT-FOLDER>. Identify any contradictions between
pages, flag topics that were mentioned but not fully explained, list any pages
that need updating, suggest wiki pages that are missing, and write a health-check
report. Save the report at <WIKI-OUTPUT-FOLDER>/_health-check-<YYYY-MM-DD>.md.
```

### Fold-back template (used in Lesson 4)

```
Read the files in <OUTPUT-FOLDER>. For each output, identify which wiki pages in
<WIKI-FOLDER> relate to it. Update those wiki pages with the distilled insights
from the outputs — do not copy-paste, rewrite in the wiki's style. Do not delete
anything from the wiki.
```

## 11 🎓 TEACHING NOTES (FOR THE INSTRUCTOR)

**Timing:** ~15 minutes. The skill does the heavy lifting; students mostly approve writes.

**Live demo:** Run `/second-brain build [area]` in front of students. Walking them through the skill's tool calls (read raw, read about-me, synthesize topics, save pages, log) makes the magic concrete. Bonus: show them where the skill file lives (`cowork-obsidian/skills/second-brain/SKILL.md`) so they understand it's just a markdown file Cowork reads.

**Emphasis:** *"Build once. Update every week. Health-check every month. Fold-back every quarter. The skill knows. You just type."*

**Watch for:**

- Students who reach for the appendix prompts and try to paste them. **Push back.** The skill runs the prompts. Pasting is duplicate work.
- Students who run `build` on an area that already has a wiki. The skill should warn them — confirm they read the warning before approving.
- Students in `external` mode who get frustrated at the per-write prompts. Offer the overlay solution as a way to "trade external-mode friction for overlay-mode silent writes on one specific sub-folder."
- Students who hand-edit the wiki and then run `update` and lose their edits. Push back: wiki is AI territory. Their edits belong in raw.
