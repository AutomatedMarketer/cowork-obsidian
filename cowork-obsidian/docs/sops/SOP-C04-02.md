## 1 🎯 WHAT THIS LESSON GIVES YOU

By the end of this lesson, one of your life areas (or one slice of your existing vault, in `external`/`overlay` mode) has a real wiki — multiple short pages, each on a single topic, cross-linked, built by Claude from raw notes. You'll also know the three prompts that work on every life area forever:

- **Build** — turn raw into a fresh wiki (first-run only, for each area).
- **Update** — keep an existing wiki in sync with new raw (safe to run anytime).
- **Health check** — surface contradictions, gaps, and stale pages so you can fix them.

Plus an optional fourth: **Fold-back** — absorb a finished output back into the wiki so the wiki keeps getting smarter.

You'll run each prompt once. After this, maintenance is a 5-minute job every week or two.

**New in v0.2.0:** the `/second-brain` skill is **mode-aware**. The same three prompts work differently depending on whether your vault is `scaffold`, `overlay`, or `external` mode. Same shape, different scoping. We cover all three modes below.

---

## 2 🧠 THE BIG IDEA (READ THIS FIRST — IT'S SHORT)

The three prompts are the only way you ever touch the wiki. You don't write pages by hand. You don't tidy them. You don't fight Claude over wording. You write into raw freely, then you run one of three prompts, and the wiki gets built or refreshed.

Think of it like a librarian's workflow:

- **Build** is opening a new section of the library for the first time — cataloguing every book in a shipment, labelling shelves, cross-referencing authors.
- **Update** is when new books arrive. The librarian integrates them, updates the card catalogue, adds cross-references. Nothing gets thrown away.
- **Health check** is the periodic walk through the stacks: *"Is anything filed wrong? Is any catalogue card out of date? Are there books referenced in the catalogue that aren't on the shelf?"*

Three prompts. Forever. You'll re-run them for every life area, week after week, and the wiki keeps compounding.

**One rule that prevents 90% of confusion: build is destructive, update is safe.** Build overwrites everything in the wiki folder. Update preserves and adds. Run build once per life area — the first time. Every refresh after that is update.

## 3 🔍 HOW MODE CHANGES BEHAVIOR

| Mode | What `build` does | What `update` does | What `health-check` does |
|---|---|---|---|
| **`scaffold`** | Reads `<life-area>/raw/`. Writes fresh wiki pages to `<life-area>/wiki/`. Standard v0.1.0 flow. | Reads new raw notes since last update. Extends or creates wiki pages. Never deletes. | Reads the entire `<life-area>/wiki/`. Reports contradictions, gaps, stale pages. |
| **`overlay`** | Same as scaffold, scoped to `<overlay-root>/raw/` → `<overlay-root>/wiki/`. | Same shape. | Same shape. |
| **`external`** | **Asks you per-run** which folder holds raw input and where to write the wiki. Default suggestion: `<chosen-input>/wiki/` as a sibling. **Every page asks permission before saving.** | Same — asks per-run, asks per-page. | Same — asks per-run which folder to audit. |

If your vault is `external` mode, plan for slightly more back-and-forth. The skill never silently creates folders or saves pages without your nod. That's by design — your existing structure is yours.

## 5 🤖 DO THE THING — THE EASY WAY (RECOMMENDED)

One paste-ready prompt that runs all three canonical prompts end-to-end. Cowork picks the life area / sub-folder based on your vault's mode, builds its first wiki, adds a new raw note to trigger update, runs update, then runs a health check. Three in one.

### Step 1 — Open Cowork at your vault root

The vault root is where your `vaults.md` reference lives (via `projects/second-brain/vaults.md` in your Cowork workspace), and Cowork needs to be able to read your `about-me/` files so identity and voice load every session.

If Cowork is pointed at your Cowork workspace from Project 01, you can stay there — `vaults.md` lives in the workspace, not in the vault. Cowork reads `vaults.md` to know where each vault lives, then reads vault content via the `safe-zones.md` carve-out we set up in Lesson 1.

### Step 2 — Start a New task and paste this prompt

```
I'm on Lesson 2 of Project 04 — Second Brain. I want to run all three canonical wiki
prompts end-to-end against one life area or sub-folder, starting from the raw notes
that are already in place.

Walk me through each part, pausing for confirmation between them.

Step A — Pick the target. Read projects/second-brain/vaults.md. Identify the
default vault and its mode.
- If mode is scaffold: list every life-area folder that has at least one file in its
  raw/ subfolder. Recommend the one with the most raw material. Ask me to confirm.
- If mode is overlay: target overlay_root automatically. Tell me what's in
  <overlay_root>/raw/ and confirm I want to proceed against it.
- If mode is external: ask me which folder holds the raw input I want you to read,
  and where I want the wiki output to go (default suggestion: <chosen-input>/wiki/).
  Confirm both before any read.

Step B — Run the Build prompt. Use this exact prompt template, substituting the
target you confirmed:

  "Read every file inside of <RAW-INPUT-FOLDER>. Create one wiki page per major topic
   you find across the files. Write each wiki page as a clean markdown file with a
   title, a two-to-three-sentence summary, key points, and cross-links to related
   pages using [[wikilink]] syntax. Save every file into <WIKI-OUTPUT-FOLDER>. Do
   not summarize multiple topics into one file — create separate pages per topic."

Execute. After it completes, list every new file you wrote with a one-line summary
each. Tell me how many pages you created and whether any raw content got condensed
into a single page that should have been split. In external mode, ask permission
before each batch of writes (10 pages max per batch).

Step C — Seed one new raw note to trigger Update. Interview me briefly: ask for one
new idea, observation, or note that belongs in this area. Save it as a new .md file
inside the raw input folder with a descriptive filename and today's date prefix
(e.g., 2026-05-09-recent-pattern-noticed.md). Show me the file before and after.

Step D — Run the Update prompt. Use this exact template:

  "Read every file in <RAW-INPUT-FOLDER> and check the wiki at <WIKI-OUTPUT-FOLDER>.
   Update any wiki pages that have new relevant information from raw, add new wiki
   pages for any topics that aren't yet covered, update all cross-links, add keyword
   links where relevant. Do NOT delete any existing pages. Only add and improve."

Execute. After it finishes, tell me exactly which pages were modified and which new
pages were created as a result of the raw note I just added. Do not modify pages
that had no reason to change.

Step E — Run the Health Check prompt. Use this exact template:

  "Read all the wikis in <WIKI-OUTPUT-FOLDER>. Identify any contradictions between
   pages, flag topics that were mentioned but not fully explained, list any pages
   that need updating, suggest wiki pages that are missing, and write a health-check
   report."

Save the report at <WIKI-OUTPUT-FOLDER>/_health-check-<YYYY-MM-DD>.md (today's date)
so I can find it later. Then print the key findings inline so I can skim them.

Step F — Save the three prompts for reuse and log the run. Save the three canonical
prompts as a single reference file at the vault root: _wiki-prompts.md, with one
section per prompt (Build, Update, Health Check), each with a description of when
to use it and the exact prompt text. Also append an entry to the vault's
about-me/memory.md (or projects/second-brain/memory.md if external mode and the
vault has no about-me) dated today, noting: vault, mode, target area, number of
pages, that the three canonical prompts have been tested.

Ask permission before each file write. In external mode, ask permission per-page.
Do not skip ahead.
```

### Step 3 — Follow Cowork's walkthrough

Five to fifteen Allow prompts depending on mode (scaffold is fewer, external is more). Answer the "new raw note" interview honestly. When the health check produces a report, read it — it's often more useful than the wiki itself on the first run.

## 6 ✅ MAKE SURE IT WORKED (3 MINUTES)

Three checks.

### Check 1 — The wiki folder has multiple pages, not one mega-page

Open your file manager (or Obsidian). Navigate to the wiki output folder.

What should happen: 3–15 separate `.md` files, each on a distinct topic. Titles visible. Each file short (a page or two max).

If you got one giant file instead → tell Cowork:

> *"The build produced one mega-file instead of separate pages. Re-run the Build prompt and emphasize that I want ONE page per topic, separate files, no summarizing multiple topics into one file. Delete the mega-file before re-running."*

### Check 2 — Cross-links exist and work

Open any wiki page in Obsidian. Look for `[[Another Page Name]]` links. Click one — it should jump to the referenced page.

If there are no cross-links → tell Cowork:

> *"My wiki pages don't have cross-links between them. Re-run the Update prompt with an explicit instruction to add cross-links between any pages that relate to each other's topics."*

### Check 3 — Health-check report and `_wiki-prompts.md` exist

- `<WIKI-OUTPUT-FOLDER>/_health-check-<today>.md` — contains a list of contradictions, missing topics, pages needing updates, and pages suggested to add.
- `_wiki-prompts.md` at the vault root — contains all three prompts saved for easy reuse.

If either is missing → tell Cowork:

> *"The [health-check / prompts] file doesn't exist. Confirm the file write actually happened and retry if not."*

The golden rule: **describe what you see, ask Claude to fix it.**

## 10 🛡 SAFETY NOTE

Two rules to keep the wiki trustworthy:

1. **Never run Build on a life area or folder that already has a wiki.** Build overwrites. If you want to refresh an existing wiki, use Update. If the wiki is truly broken and you want to rebuild, copy the existing `wiki/` folder to `wiki-backup-<date>/` first, THEN run Build. That way you can recover.

2. **Back up the vault.** It's a folder of `.md` files — commit it to a private Git repo (Lesson 3 covers this), zip it weekly, or use any backup tool. If a bad prompt ever scrambles your wiki (rare but possible), a backup means 60 seconds of recovery instead of starting over.

## 11 🔍 UNDER THE HOOD — FOR THE CURIOUS (TOTALLY OPTIONAL)

Skip this unless you want to know why three prompts and not six.

### Why exactly three

The three prompts map to three distinct operations:

- Create something new (**Build**)
- Synchronize old with new (**Update**)
- Audit what exists (**Health Check**)

Any second-brain system you'll ever need can be expressed as some combination of those three. Adding more prompts (e.g. "summarize," "restructure," "compress") is usually a variation that fits inside one of the existing three — *"run a health check and flag anything too long; then ask Update to split those pages"* rather than a fourth prompt.

Three is the minimum that covers the workflow. More prompts = more cognitive overhead = people stop running them.

### Why mode-awareness matters

In `scaffold` mode the paths are predictable: `<life-area>/raw/` → `<life-area>/wiki/`. The skill can run silently. In `external` mode the paths are whatever you chose — the skill can't guess, so it asks. The trade-off is friction for safety.

If you find external mode too slow for a frequent target, you can always create an `overlay` for that one slice (re-run `/onboard-second-brain` and add an overlay on the sub-folder you care about). Then operations on that sub-folder become silent again.

### Why the wiki is separate from raw

Your raw notes are stream-of-consciousness. Dates attached, emotional context, mid-thoughts, tangents. They're valuable as primary sources but terrible to scan when you want to remember something.

The wiki is a second-pass representation of the same material — organized, cross-referenced, clean. Claude's the right tool for this because it can hold the whole raw corpus in mind, spot patterns, and produce consistent structure. You're the right tool for raw because only you have your original thoughts.

Separation of concerns: you write what matters, Claude organizes what you wrote. Neither tries to do both.

### What `[[wikilinks]]` actually are in Obsidian

A cross-link is `[[Page Name]]` inside any markdown file. Obsidian renders these as clickable links. Click → jump to that page. If the page doesn't exist yet, the link is dim; click it and Obsidian offers to create the page.

Over time, your wiki becomes a graph of interconnected pages. Obsidian's graph view visualizes it. Claude can traverse the cross-links when you ask questions — *"what do I know about X?"* becomes a walk through the linked pages.

### The `_` prefix on health-check filenames

`_health-check-2026-04-21.md` starts with an underscore so it sorts to the top or bottom of alphabetical listings (depending on OS), separating "system files" from "content files." Same trick applies if you later add `_index.md`, `_todo.md`, etc. Not required — just convenient.

## 12 🎓 TEACHING NOTES (FOR THE INSTRUCTOR)

**Timing:** 30 minutes for scaffold/overlay students. 40–45 for external students because of the per-write permission prompts.

**Live demo:** Build a wiki live from 5 seed notes. Watching Claude identify 3–4 topics from those notes and create cross-linked pages is the "aha" of the whole project.

**Emphasis:** *"Build once. Update every week. Health check every month."* That's the entire maintenance model.

**Watch for:**

- Students who start hand-editing the wiki. Remind them: wiki is AI territory. Their edits belong in raw.
- Students who run Build a second time and create duplicates. Explicitly forbid running Build twice on the same area.
- Students in `external` mode who skip the persona picker. Run it with them — it makes the prompts land in their actual work, not abstract theory.
- Students who skip the health check because "it's fine." Make them run it — the first health check usually surfaces 3–5 useful issues they didn't know about.
