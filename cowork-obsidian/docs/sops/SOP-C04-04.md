## 2 🧠 THE BIG IDEA (READ THIS FIRST — IT'S SHORT)

Lesson 2 proved the operations work. This lesson makes sure you'll actually run them for the next ten years.

Three patterns separate people whose second brain compounds from people whose second brain rots:

1. **A rhythm.** Not a strict calendar. A rule of thumb: *"if I've added 5+ raw notes to this area, it's time for an update."* Plus a real weekly reminder on your calendar so you don't forget.
2. **A home index.** Past five wiki pages, you can't keep the structure in your head. A `_index.md` at the top of each wiki fixes that.
3. **A daily-startup pattern.** When you sit down to work on the vault, you run **one command** — `/open-vault`. It launches Obsidian, preps Cowork with vault context, and tells you what to do next based on activity since last session. No reaching for the right command in your head.

There's a fourth pattern — **folding outputs back into the wiki.** Outputs you produce in Cowork sessions are better than wiki pages because they're the result of you and Claude thinking together. Periodically you want those insights merged back into the wiki, so the wiki keeps getting smarter. Quarterly is enough.

## 5 🔁 THE DAILY-STARTUP PATTERN (NEW IN v0.3.0)

This is the core insight from Lesson 4: **you don't need to remember the right command for what you want to do.** Your single daily-startup command is:

```
/open-vault
```

Or just type *"open my vault"*. The `/open-vault` skill does:

1. Loads `vaults.md` and picks the right vault (asks if you have multiple).
2. Reports vault status — last activity, last build / update / health-check, count of new raw notes since last update.
3. Launches Obsidian visually to that vault via `obsidian://open?vault=<name>` URL scheme.
4. Offers a next-step menu: build, update, health-check, fold-back, just-browse.

**That's the daily-driver.** Over time, this becomes muscle memory: sit down → `/open-vault` → it tells you what's pending → run the operation it suggests.

If `/open-vault` says *"7 new raw notes since last update"* on a Sunday, you know to run `/second-brain update <area>` next. If it says *"last health-check was 35 days ago"* on a Saturday, you know to run `/second-brain health-check <area>`.

The skill carries the cadence so you don't have to.

## 9 🛡 SAFETY NOTE

Three rules to keep the system from rotting:

1. **Stop hand-editing the wiki.** If you catch yourself wanting to "just fix a page," write the correction as a note in `raw/` instead. Then `/second-brain update` propagates it. This preserves the raw/wiki separation and means you don't have to track manual edits.

2. **Back up before every `fold-back`.** `fold-back` is the one operation that modifies existing wiki pages. Before running it, copy each area's `wiki/` folder to `wiki-backup-<date>/` first. If anything goes sideways, you can restore in 60 seconds. (If you set up Git/GitHub sync in Lesson 3, you already have automatic backups — every commit is a snapshot.)

3. **Don't skip the weekly rhythm.** The system is designed to run weekly. If you miss 6 weeks, the first `update` after the gap takes 3× as long and the quality drops. The reminder is on your calendar for a reason.

## 10 🔍 UNDER THE HOOD — FOR THE CURIOUS (TOTALLY OPTIONAL)

Skip this unless you want to know why the rhythm matters.

### Why weekly and not daily

Daily is too frequent — most raw notes don't move the needle individually. You'd run `update` every day and the wiki would churn without getting meaningfully better. Weekly lets ~5 new raw notes accumulate. Those 5 notes usually form a theme, and that theme is what becomes a new wiki page or a meaningful update.

Monthly for `health-check` because contradictions and gaps take time to emerge. Weekly would surface noise. Monthly gives you real drift signals.

Quarterly for `fold-back` because outputs need to accumulate before they're worth merging. You might produce one good output in a week; you produce ten in a quarter. Ten is meaningful; one is noise.

### Why the underscore prefix on `_index.md`

Filesystems sort alphabetically. Underscore comes before letters in ASCII, so `_index.md` sorts to the top of the Obsidian file tree (and usually the file manager). Always the first thing you see when you open a wiki folder. Same trick works for `_health-check-*.md` and any "system file" you want separated from "content files."

### Why we dropped the v0.1.0 custom-skill build

In v0.1.0, the cohort SOP-C04-03 had students build a "Vault Wiki Workflow" skill via Skill Creator. That made sense before the plugin existed — students were assembling the operational logic themselves.

In v0.3.0 the plugin already installs `/second-brain` (which IS the Vault Wiki Workflow) and `/open-vault` (which adds session prep). Building another skill that does the same thing is duplicate work and an unnecessary surface to maintain. So we dropped it.

If a student wants to learn how to build a custom Cowork skill *as a learning exercise*, they can use Skill Creator independently — but for Project 04, the plugin's skills are enough.

### How `/open-vault` differs from `/second-brain`

`/open-vault` is read-only state-prep + launching Obsidian. It tells you what's going on. It never builds, updates, or modifies the vault.

`/second-brain` is the worker. It modifies the wiki (with permission) when you tell it to.

You run `/open-vault` first (to load context and see status), then `/second-brain <operation> <area>` second (to act on what `/open-vault` showed you).

## 11 🎓 TEACHING NOTES (FOR THE INSTRUCTOR)

**Timing:** ~20 minutes. The calendar event creation is the slowest part (3 events × confirm each).

**Live demo:** Show the daily startup. Run `/open-vault` in a populated vault. Walking students through the skill's status report (X raw notes new, last update was Y) is the moment they understand why this command exists.

**Emphasis:** *"Rhythm beats enthusiasm. Weekly for ten years beats daily for three weeks. And `/open-vault` is the one command you remember."* That's the line.

**Watch for:**

- Students who ask why we don't build a custom skill like in the v0.1.0 SOPs. Answer: the plugin already provides the operations. Custom skill is duplicate.
- Students who skip the calendar reminders. Force them to do it live — by the time they "remember tomorrow" they won't.
- Students whose `/open-vault` doesn't launch Obsidian visually. Walk them through the URL-scheme fallback (file manager). Don't fail the lesson on that — Cowork session prep is the more valuable half.
- Students who want to run `fold-back` weekly "to keep it fresh." Push back — quarterly. Too frequent damages the wiki.
- External-mode students who try to write `My Vault System.md` into their vault. Reroute to `projects/second-brain/memory.md` per the hard rule.
