## 1 🎯 WHAT THIS LESSON GIVES YOU

By the end of this lesson your second brain has:

- A **rhythm** — daily raw, weekly update, monthly health check, quarterly fold-back.
- A **home index** for navigating any wiki that's grown past ten pages.
- One **golden output** saved and referenceable.
- A **custom slash-command skill** that runs all three canonical prompts (plus fold-back) without you typing the prompts by hand.
- **Calendar reminders** so the system runs itself instead of relying on memory.
- The `**My Vault System.md**` file you intentionally skipped in Lesson 1 — written now that you've actually used the vault.

You'll build all of this by pasting one prompt into Cowork. The skill build alone would take 30 minutes of hand work — Cowork does it in five while you watch.

---

## 2 🧠 THE BIG IDEA (READ THIS FIRST — IT'S SHORT)

Lesson 2 proved the prompts work. This lesson makes sure you'll actually run them for the next ten years.

Three patterns separate people whose second brain compounds from people whose second brain rots:

1. **A rhythm.** Not a strict calendar. A rule of thumb: *"if I've added 5+ raw notes to this area, it's time for an update."* Plus a real weekly reminder on your calendar so you don't forget.
2. **A home index.** Past ten wiki pages, you can't keep the structure in your head. A `_index.md` at the top of each wiki fixes that.
3. **A skill that runs the prompts for you.** Typing the canonical prompts every week is death by friction. A skill turns three prompts into three slash commands.

There's also a fourth pattern — **folding outputs back into the wiki.** The outputs you produce in Cowork sessions are better than wiki pages because they're the result of you and Claude thinking together. Periodically you want those insights merged back into the wiki, so the wiki keeps getting smarter. Quarterly is enough.

## 3 🤖 DO THE THING — THE EASY WAY (RECOMMENDED)

One prompt. Cowork:

1. Adds a home index to each life area's wiki (or the wiki of each overlay sub-folder, or the external-mode wiki paths you've used).
2. Saves one golden output from a recent Cowork conversation (or invents a useful starter if you don't have one yet).
3. Builds the **Vault Wiki Workflow** skill that wraps Build / Update / Health Check / Fold Back into a slash command.
4. Creates three calendar events to establish your rhythm (weekly update, monthly health check, quarterly fold-back).
5. Writes the `My Vault System.md` file you left blank in Lesson 1 (only for `scaffold` mode — `external` mode users skip this; `overlay` mode users get a tighter version).

### Step 1 — Open Cowork

Cowork should already be set up to read your vault from Lesson 1's Phase 4 (the safe-zones carve-out). If your folder pill doesn't read your vault, fix it: Settings → Cowork → change folder → pick your vault root (or your Cowork workspace if `vaults.md` lives there).

### Step 2 — Start a New task and paste this exact prompt

```
I'm on Lesson 4 of Project 04 — Second Brain. My vault already has one or more
life areas (or sub-folders for overlay mode, or external folders) with a functional
wiki from Lesson 2. Now I want to set up the maintenance layer: a home index per
wiki, a saved golden output, a custom skill that runs all the canonical prompts,
recurring calendar reminders for the rhythm, and the My Vault System.md file at
the vault root (scaffold mode only).

Read projects/second-brain/vaults.md first to know which vault and which mode I'm
working with. Walk through each step, pausing for my confirmation.

Step A — Home index for every wiki.
- For scaffold mode: list every life area whose wiki has at least 5 pages.
- For overlay mode: target the overlay's wiki only.
- For external mode: ask me which folder(s) hold wikis I want indexed.

For each one, generate a _index.md file inside the wiki folder. The index lists
every wiki page grouped by theme, with a one-line summary of each, and cross-links
for navigation. Do NOT modify the other wiki pages — only create or update
_index.md. Use the underscore prefix so the file sorts to the top of the Obsidian
file tree.

Show me one example index inline so I can read its format before saving the rest.

Step B — Save a golden output. Ask me: "What's one thing you and I have worked on
recently that you want preserved as a golden output? If you can't think of one,
I'll invent a useful starter — for example, a 'Second-Brain Operating Manual'
summarizing how I maintain this vault."

Then:
- If I name something specific: save it as <LIFE-AREA-or-FOLDER>/output/<YYYY-MM-DD>-<short-title>.md
  with a one-line summary at the top and the content below. Ask which area.
- If I say "invent one": write a concise Second-Brain Operating Manual pulling
  from the lessons I've completed — the raw/wiki/output model, the three canonical
  prompts, my cadence, and my rhythm. Save it to about-me/Second-Brain Operating
  Manual.md (scaffold/overlay) or to a path I name (external).

Step C — Build the Vault Wiki Workflow skill. Use the built-in Skill Creator (or
write the skill file directly if Skill Creator is too slow). The skill should:

  1. Ask me which vault to work on (read vaults.md, list configured vaults, ask
     me to pick if more than one).
  2. Ask me which life area / overlay sub-folder / external folder to target.
     Dynamically derive this from the vault's configuration — do NOT hardcode.
  3. Ask me which operation: BUILD, UPDATE, HEALTH-CHECK, or FOLD-BACK.
  4. Based on my answers, run the exact canonical prompts from _wiki-prompts.md
     at the vault root (or wherever I saved them in Lesson 2). Apply mode-aware
     scoping: scaffold = standard, overlay = overlay-root scoped, external = ask
     per-write.
  5. Also include a FOLD-BACK prompt with this template:

     "Read the files in <LIFE-AREA-or-FOLDER>/output/. For each output, identify
      which wiki pages in <WIKI-FOLDER> relate to it. Update those wiki pages with
      the distilled insights from the outputs — do not copy-paste, rewrite in the
      wiki's style. Do not delete anything from the wiki."

     Save FOLD-BACK as a fourth section inside _wiki-prompts.md.

  6. Before running any operation, always load context from about-me/ (vault's or
     workspace's) so the output matches my voice and working style.
  7. After running, append a one-line entry to memory.md (in the vault's
     about-me/ for scaffold/overlay, or projects/second-brain/memory.md for
     external) with date, vault, mode, target, operation performed.

Save the skill and tell me how to invoke it (slash command if available, otherwise
by natural-language name). Show me the skill file path.

Step D — Create rhythm reminders on Google Calendar. Create three recurring events:
  - "Second Brain — weekly update" every Sunday 5 p.m., 15 minutes,
    notes: "Run the Vault Wiki Workflow skill → UPDATE on any area with 5+ new raw
    notes this week."
  - "Second Brain — monthly health check" first Saturday of each month, 20
    minutes, notes: "Run HEALTH-CHECK on every area. Read each report. Drop any
    clarifying notes into raw/ so UPDATE will fix issues automatically."
  - "Second Brain — quarterly fold-back" first Sunday of January, April, July,
    October, 30 minutes, notes: "Run FOLD-BACK on every area that has outputs
    accumulated. Refresh every _index.md. Review stale pages."

Show me each event before creating and ask permission.

Step E — Write the My Vault System.md file (scaffold/overlay only — skip for
external).

This was intentionally skipped in Lesson 1. Now that I've used the vault, write it.
Interview me briefly (2–3 questions max): what life areas have I set up, which
ones are most active, and anything specific about how I want you to navigate my
vault. Draft the file in 15–25 lines — references the raw/wiki/output model, lists
my life areas (or overlay sub-folder), names the three canonical prompts, and says
"always load about-me/ first." Save it at about-me/My Vault System.md inside the
vault.

For external mode: instead of writing My Vault System.md, append a 5-line "How to
work in this vault" section to projects/second-brain/memory.md so the operational
context is preserved without writing into the user's existing vault structure.

Pause between each step. Ask permission before file writes. Skip Step B path 1 if
I can't think of a specific output — just do Step B path 2.
```

### Step 3 — Follow Cowork's walkthrough

You'll get eight to ten Allow prompts. The skill build is the biggest — it involves multiple file writes and maybe a back-and-forth with Skill Creator. Don't skip steps.

## 4 ✅ MAKE SURE IT WORKED (3 MINUTES)

Four checks.

### Check 1 — Every wiki has a `_index.md`

Open Obsidian. Click into each life area's wiki folder (or your overlay's wiki folder, or the external wiki paths). At the top of the file list you should see `_index.md`. Open one — it should list every page in that wiki with one-line summaries and working cross-links.

### Check 2 — At least one golden output exists

Navigate into one of your area's `output/` folders — or into `about-me/` if Cowork created the Second-Brain Operating Manual there. You should see at least one `.md` file with content you'd actually want to reference later.

### Check 3 — The Vault Wiki Workflow skill runs end-to-end

Start a new task. Type `/` and look for **Vault Wiki Workflow**, or just ask in natural language:

> *"Use the Vault Wiki Workflow skill — run an UPDATE on [your most-active life area]."*

Cowork should:

- Load your `about-me/` files first (you'll see file-read tool calls).
- Ask which vault and which area/operation if you didn't specify.
- Run the correct canonical prompt with mode-aware scoping.
- Append a log entry to `memory.md` after the run.

If the skill doesn't show up → tell Cowork:

> *"I can't find the Vault Wiki Workflow skill. Tell me the skill file's absolute path and confirm it's in the right location for Cowork to see it. If it's in the wrong place, move it."*

### Check 4 — The three calendar events are on Google Calendar

Open Google Calendar. Check:

- Sunday 5 p.m. weekly — Second Brain weekly update.
- First Saturday of next month — Second Brain monthly health check.
- First Sunday of January/April/July/October — Second Brain quarterly fold-back.

## 8 🛡 SAFETY NOTE

Three rules to keep the system from rotting:

1. **Stop hand-editing the wiki.** If you catch yourself wanting to "just fix a page," write the correction as a note in `raw/` instead. Then UPDATE propagates it. This preserves the raw/wiki separation and means you don't have to track manual edits.

2. **Back up before every FOLD-BACK.** Fold-back is the one operation that modifies existing wiki pages. Before running it, copy each area's `wiki/` folder to `wiki-backup-<date>/` first. If anything goes sideways, you can restore in 60 seconds. (If you set up Git/GitHub sync in Lesson 3, you already have automatic backups — every commit is a snapshot.)

3. **Don't skip the weekly rhythm.** The system is designed to run weekly. If you miss 6 weeks, the first UPDATE after the gap takes 3× as long and the quality drops. The reminder is on your calendar for a reason.

## 9 🔍 UNDER THE HOOD — FOR THE CURIOUS (TOTALLY OPTIONAL)

Skip this unless you want to know why the rhythm matters.

### Why weekly and not daily

Daily is too frequent — most raw notes don't move the needle individually. You'd run Update every day and the wiki would churn without getting meaningfully better. Weekly lets ~5 new raw notes accumulate. Those 5 notes usually form a theme, and that theme is what becomes a new wiki page or a meaningful update to an existing one.

Monthly for health check because contradictions and gaps take time to emerge. Weekly would surface noise. Monthly gives you real drift signals.

Quarterly for fold-back because outputs need to accumulate before they're worth merging. You might produce one good output in a week; you produce ten in a quarter. Ten outputs is a meaningful update to the wiki; one output is noise.

### Why the underscore prefix on `_index.md`

Filesystems sort alphabetically. Underscore comes before letters in ASCII, so `_index.md` sorts to the top of the Obsidian file tree (and usually the file manager) — always the first thing you see when you open a wiki folder. Handy. Same trick works for `_health-check-*.md` files, `_wiki-prompts.md`, and any "system file" you want separated from "content files."

### Why Fold-Back is quarterly, not automatic

Two reasons:

1. **Outputs are rare.** If you're doing real work with Claude, you produce maybe 5–10 good outputs per quarter. Auto-running fold-back weekly would usually do nothing useful.
2. **Fold-back modifies existing wiki pages.** That's the one destructive-ish operation in the whole system. Forcing it to be a deliberate, calendared action (not automatic) keeps you in the loop when it runs.

Always backup before fold-back. Always.

### How Skill Creator differs from manual skill building

Skill Creator is a guided, evaluated approach. It asks probing questions, writes a skill file, then generates test cases to verify the skill works. Thorough, but slow — 10–30 minutes for a single skill and meaningful token usage.

For a skill as well-scoped as Vault Wiki Workflow (three prompts, one router), you can sometimes shortcut: *"Don't use Skill Creator. Just write a simple skill.md directly at the right path with these exact contents: [...]. No evals, no test suite."* The result works at 10% of the token cost.

Skill Creator earns its cost for complex skills (multi-step research, code generation, tasks where the prompt shape isn't obvious). For simple routers, consider skipping it.

### Where the skill lives on disk

Depending on how Skill Creator saved it:

- **Global, cross-workspace:** `~/.claude/skills/vault-wiki-workflow/skill.md` (Mac/Linux) or `%APPDATA%\Claude\skills\vault-wiki-workflow\skill.md` (Windows).
- **Workspace-scoped:** `<workspace-root>/.claude/skills/vault-wiki-workflow/skill.md`.

Let Cowork tell you the exact path. Edit directly when you add a life area / vault / overlay.

### How mode-awareness changes the skill

In v0.1.0 the skill assumed scaffold mode only. In v0.2.0 the skill reads `vaults.md`, identifies the active vault's mode, and applies the right scoping:

- `scaffold` → operates on `<vault>/<life-area>/{raw,wiki,output}/` paths automatically.
- `overlay` → operates on `<overlay-root>/{raw,wiki,output}/` paths automatically.
- `external` → asks the user per-run which folders to read from and write to. Caches the last-used answer per vault.

If your skill doesn't behave this way, the skill file is from v0.1.0. Re-run Step C of Section 3 to regenerate it.

## 10 🎓 TEACHING NOTES (FOR THE INSTRUCTOR)

**Timing:** 30 minutes. Skill Creator dominates — half the lesson is waiting for Skill Creator to finish. If students are tight on time, offer the shortcut path (skip Skill Creator, write `skill.md` directly).

**Live demo:** show the "before" (typing the Build prompt by hand) vs. "after" (slash command with the skill). The friction difference is the whole sell.

**Emphasis:** *"Rhythm beats enthusiasm. Weekly for ten years beats daily for three weeks."* That's the line.

**Watch for:**

- Students who skip the calendar reminders. Force them to do it live — by the time they "remember tomorrow" they won't remember.
- Students whose skill build takes forever. Offer the shortcut path (skip Skill Creator, write `skill.md` directly).
- Students who want to run FOLD-BACK weekly "so it always stays fresh." Push back — quarterly. Too frequent damages the wiki.
- External-mode students who get frustrated by per-run prompting. Offer the caching tweak (cache last-used input/output paths per vault) — it solves the friction without compromising the per-write safety.
